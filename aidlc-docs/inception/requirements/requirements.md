## 1. ビジネス意図（Intent）

### 1.1 プロジェクトの本質的なコンセプト

「恋愛自動操縦」は、マッチングアプリにおける会話をAIが全自動で設計・支援するサービスである。

**背景にある現状の課題：**
マッチングアプリで出会い、メッセージを重ねても、結局は同じような会話の繰り返し。そこに疲弊して、会えなくなるのは勿体ない。本サービスはその課題を解決するために生まれた。

**「人をダメにする」との接続：**
しかし、立ち止まって考えてほしい。恋愛とは本来、個と個が試行錯誤しながら歩み寄るプロセスである。その過程で生まれる緊張・失敗・喜びこそが人間的な成長につながる。その大切なプロセスをAIに全部やらせることは、相手の気持ちを踏み躙る倫理的に問題のある行為である。

それこそが、人を本当の意味でダメにするサービスである。さらに、自分のメッセージが他ユーザーにコピーされることで、他人の思考力まで奪っていく連鎖構造を持つ。便利であればあるほど、ユーザーは自分で考えなくなる。

**キャッチコピー：**
> 「もう悩まなくていい。あなたの恋愛、AIが操縦します。」

### 1.2 ターゲットユーザー

- マッチングアプリを使っているすべての人（年齢・性別問わず）
- 特に「最初のメッセージが苦手」「会話が続かない」「なかなか会えない」と感じているユーザー

### 1.3 解決する課題

| 既存サービスの課題 | 本サービスの解決策 |
|---|---|
| その場その場の返信案しか出ない（戦略がない） | ゴールから逆算した会話ルート全体を設計 |
| 自分らしさが出せない（テンプレ感が強い） | 自分のキャラクター設定をもとにパーソナライズ |
| 結局会いにくい（目的達成率が低い） | ゴール（電話 / ご飯）を明示して最短ルートを提示 |
| 成功パターンが蓄積されない | 実績データをランキング化して再利用可能に |
| 相手のことを深く理解できない | AIが相手プロフィールを分析して人物像を推定 |

## 2. 機能要件

### 2.1 F-001: キャラクター設定

ユーザーが自分のプロフィール・話し方・目標を事前登録する。登録した設定はすべてのメッセージ生成に反映される。

**登録項目：**
- 話し方のトーン（フランク / 丁寧 / 面白い系 など）
- 自分の趣味・職業・年齢などの基本情報
- 得意なデートプランや行きたい場所
- 過去の成功パターン（うまくいったメッセージの傾向など）

### 2.2 F-002: 相手情報入力

相手の趣味・年齢・プロフィール情報を入力する。

**入力項目：**
- 相手の年齢
- 相手の趣味・興味
- 相手のプロフィール文（任意）

### 2.3 F-003: 相手ペルソナ自動分析【新機能】

相手のプロフィール文をAIが分析し、人物像・性格傾向・刺さるアプローチを自動推定する。

**要件：**
- Amazon Bedrockを使用してプロフィール文を分析
- 性格傾向・価値観・コミュニケーションスタイルを推定
- 分析結果をメッセージ生成・会話ルート設計に自動反映
- 分析結果をユーザーに可視化して表示

### 2.4 F-004: ゴール設定

会話のゴールを設定する。

**選択肢：**
- 電話につなげたい
- ご飯に行きたい

### 2.5 F-005: 最初のメッセージ生成

相手情報 + ペルソナ分析 + キャラクター設定 + ゴールをもとに、AIが最初のメッセージを生成する。

**要件：**
- テンプレ感・AI感を排除したパーソナライズされた文章を生成
- Amazon Bedrock（Claude）を使用
- 複数パターンを提示し、ユーザーが選択できる

### 2.6 F-006: 会話ルート生成

最初のメッセージから、ゴールまでの会話シナリオを一覧で提示する。

**要件：**
- 相手の返信パターンを予測し、複数のルートを提示
- 各ルートにゴール達成難易度を表示
- 「好反応ルート」「慎重ルート」など複数シナリオを提示

### 2.7 F-007: 返信対応（会話の練り直し）

実際に相手から来た返信を入力し、AIが次のメッセージを再生成する。

**要件：**
- 現在の会話状況を分析してゴールに向けた次のメッセージを生成
- 方針変更（ゴールの変更）にも対応
- どのステップで返信が止まったかを記録する

### 2.8 F-008: 実績記録・振り返りレポート【新機能】

実績データを収集し、会話終了後にAIが振り返りレポートを自動生成する。

**要件：**
- 返信が来たかどうかを記録（どのメッセージで止まったかも記録）
- 「会えた！」ボタンで会えたかどうかを自己申告
- ポイント獲得・バッジなどのゲーミフィケーション演出
- 会話終了後の振り返りレポート自動生成
  - 返信率・ゴール達成状況の分析
  - 改善点の提示
  - 次回推奨戦略の提案

### 2.9 F-009: 舐めプスコア【新機能】

使えば使うほどスコアが上昇する、「AIに舐めプしながら恋愛する」を数値で体験させる機能。

**スコア設計：**

| 軸 | 名称 | 加算タイミング | 方向 |
|---|---|---|---|
| 軸1 | 舐めプ度 | AIを使うたびに加算 | 上昇（上限なし） |
| 軸2 | 指名数 | 自分のメッセージが他ユーザーにコピーされるたびに加算 | 上昇（上限なし） |

**要件：**
- スコアは数値のみで表示
- 舐めプ度・指名数を別々に表示
- 指名ランキングは指名数が多いほど上位
- 舐めプ度ランキングはAI使用回数が多いほど上位

### 2.10 F-010: 指名ランキング機能

実績データ・スコアをもとにランキングを表示する。

**要件：**
- 自分の過去実績ランキング（プライベート）
- 全ユーザーの匿名化された実績ランキング（全体公開）
- 利用開始時に同意を取得する
- ランキング対象：
  - 指名ランキング（指名数が多い順）
  - 舐めプ度ランキング（AI使用回数が多い順）
  - 返信率が高いメッセージランキング
  - 会えたルートランキング
- 個人を特定できる情報は含めない（匿名化・統計化）

## 3. 非機能要件

### 3.1 技術スタック

| 項目 | 内容 | 理由 |
|---|---|---|
| フロントエンド | Next.js（TypeScript） | Webアプリ。モダンで開発効率が高い |
| AIエンジン | Amazon Bedrock（Claude） | 生成AIによるパーソナライズが本サービスのコア価値。「AIが人をダメにする」テーマと直結 |
| バックエンド | AWS Lambda + API Gateway | サーバーレス構成のため初期インフラコストが低く、スモールスタートに適している |
| データベース | Amazon DynamoDB | サーバーレスと相性が良い。スキーマレスで柔軟 |
| 認証 | Amazon Cognito | ユーザーデータ管理に必要 |
| ホスティング | AWS Amplify | フロントエンドの簡易デプロイ |
| IaC | AWS CDK（TypeScript） | インフラをコードで管理することで再現性・保守性を確保する |

### 3.2 対象マッチングアプリ

**汎用設計**（特定アプリに依存しない）

### 3.3 プロジェクト制約

- 成果物：Inceptionフェーズドキュメント + 動作するプロトタイプ
- 公開：GitHubリポジトリ（公開）

### 3.4 セキュリティ

- セキュリティ拡張：**無効**（PoC・プロトタイプのため）
- 個人の会話データは匿名化して扱う
- ランキング利用時は同意取得を必須とする

### 3.5 テスト方針

- プロパティベーステスト：**スキップ**（プロトタイプフェーズのため最小限のテストで可）

## 4. ビジネス価値との対応

| 観点 | 対応内容 |
|---|---|
| ビジネス意図の明確さ | 「恋愛の思考をAIに丸投げ = 人をダメにする」という明確なコンセプト。既存サービスとの差別化も明示 |
| 創造性とテーマ適合性 | 舐めプスコア（AIに任せるほど上昇）・指名ランキング（ホスト・キャバ文化を借用）・他人の思考力を奪う連鎖構造 |
| Unit分解の適切さ | 9 Unitに適切に分割。各Unitが独立した責務を持つ |
| ドキュメントの品質 | AIDLCワークフローに従った体系的なドキュメント整備 |
